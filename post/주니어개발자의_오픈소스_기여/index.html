주니어개발자, 오픈소스에 처음으로 기여해보기 까지

By on January 07, 2021

오픈소스에 기여 한다는 것은 오픈소스 기여 경험이 없는 개발자들에게는 슬프게도 상당히 이상적인 일입니다.
저같은경우엔 어떤 프로젝트에, 무엇을, 어떻게 해야할지 모든 단계가 부담스러웠습니다. 먼 이야기라고 생각하며 오픈소스에 기여해본 사람들은 범접할 수 없는 개발자들 일 것이라고 막연히 생각했습니다.
그래서 기여는 언젠간 해볼 숙제 같은 존재였는데요,

그랬던 제게 정말 우연히! 쉽게!
그것도 spring boot를 사용하는 개발자들은 한번쯤 들어봤을 법한 오픈소스인 querydsl 에 기여하게 되었습니다.

제가 기여한 경우는 ‘코드 개선’ PR이었는데요, 솔직히 정-말 작은 수정이었습니다.
해당 코드는 querydsl을 공부하다가 우연히 하게 되었습니다.
오픈소스 기여가 주 목적이 아니었던 것이죠!

솔직히 제가 개선했다고 하기엔 상당히 민망하긴 합니다만, 그럼에도 불구하고 그 과정안에서 많은 깨달음과 배움이 있었습니다.
(그래도 해당 코드… 누군가는 수정했을 거라고 생각합니다ㅎㅎ)

어쩌다 발견했는지, 그래서 어떻게 기여하게 됐는지 그 과정을 공유해봅니다 :)

Querydsl 도입 시도

querydsl 을 팀 전체 프로젝트에 도입하는 것이 제 궁국적 목표였습니다. 작은 프로젝트에 querydsl을 도입해본 후, 엄청난 매력을 느꼈기 때문입니다.
(tmi: 팀의 프로젝트들은 springboot + ORM(jpa, criteriaQuery) 사용 중)

최소한 제가 소속된 팀에는 적용하고 싶었기 때문에, 팀 내 가장 큰 프로젝트(이하 A프로젝트)에 도입이 가능한지 먼저 확인이 필요했습니다.

설정 과정에서 문제점 발견

A프로젝트 에 querydsl 설정을 마친 후, QClass가 생성되길 기대했습니다. 하지만 QClass는 어디에도 생성되지 않았습니다.

아무래도 A프로젝트의 Entity 관리 방식 때문인 것으로 추측이 되었습니다.
(entity를 외부 저장소에 모듈화하여 저장하고, 프로젝트에 외부 모듈로 의존성 추가하여 사용)

많은 구글링(우리들의 선생님)을 해보았지만 속 시원히 알려주는 글이 없었고, 궁지에 몰린 저는 직접 querydsl 코드와 마주하길 결심했습니다.(…)

해당 코드를 확인함으로써 알고자 했던 부분은 확인이 되었습니다!

하지만 이왕 코드를 뜯어보기 시작한거 메소드에서 리턴되는 DefaultConfiguration의 내용까지 알고자 DefaultConfiguration 클래스 코드를 확인하게 되었습니다.

그러던중 IDE(intelliJ)가 친절히 warnig을 뿜어내는 코드를 발견하게 되었는데요.

심지어 이 warning은 IDE가 수정 내용까지 추천해주었습니다.
추천 내용은 다음과 같습니다.

Redundant boxing inside 'Boolean.valueOf(xxx)'

한글로 어설프게 번역해보자면 대충 boxing 이 불필요하게 발생하는 것 같습니다.

여기서부터 해당 warning과 관련된 사항들을 공부했습니다.

  • Boolean.valueOf()
  • Boolean.parseBoolean()
  • boxing/unboxing

해당 과정을 통해 원인은, boxing 했다가 다시 unboxing 되는 코드 (boolean -> Boolean -> boolean) 라는 것을 알게되었습니다!

‘아, 이거 PR 한번 날려볼까?’

이해를 완전히 하게 된 순간, PR해보고싶다는 생각이 들었습니다.

하지만.. 커밋과 PR에 앞서, 제가 한 역할이 너무나도 작게 느껴졌습니다.
사실 보시다시피 제가 한 일이라곤 warning을 발견한 것 뿐이니까요.

작은 걱정을 뒤로한 채 PR요청을 한 이유는,

  1. 수정하지 않아야 할 이유가 없다.
  2. 누군가는 언젠간 수정할 코드다.
  3. 기여로 인해 묘한 소속감이 생길 것 같았다.
  4. 앞으로의 ‘오픈소스 기여’에 한 발자국이 될 것이다.

그렇게 두근거리는 마음으로 PR을 했습니다. 👏👏👏

오픈소스에 PR을 날려본 경험이 없었기 때문에 구글링을 통해 방법을 알게 되었습니다. 해당 과정은 오픈소스 입문을 위한 아주 구체적인 가이드를 참고하였습니다 : )

여차저차 커밋 메세지를 영어로 작성하고, 다음과 같은 내용으로 PR을 날렸습니다.

Change  'Boolean.valueOf()' to 'Boolean.parseBoolean()'

- These variables are not used as 'Boolean' Object.
- Java provides Autoboxing and Unboxing, but this is an unnecessary process.

PR하고약 2주간 제가 날린 PR엔 먼지가 쌓여갔는데요….
(그 기간동안 괜히 작아지는 기분이었습니다..)

다행히 존버(?) 후 제 PR이 Merged 되었다는 것을 알게되었습니다!

드디어 첫 기여가 된 순간입니다…
저 merge 마크가 너무나도 뿌듯하네요 :>

겨우 이정도 수준의 기여?

커밋 내용을 보시다시피 정말 별 것 아닙니다.
심지어 IDE가 고치라고 알려준 코드를 PR만 날렸을 뿐입니다.

그래도, 개인 github에 붙은 querydsl 기여 딱지는 여러가지로 저에게 긍정적인 영향을 주었습니다.

  1. 오픈소스는 누구나 개선할 수 있다.
    • 간단한 개선이라도 오픈소스의 퀄리티를 더 높일 수 있다면 누구나 기여할 수 있습니다. 심지어 저처럼 IDE가 고쳐준 코드일지라두요 … : ) 해당 코드는 아무도 열어보거나 발견하지 않았다면 그대로 방치되었을 수 있는 코드라고 생각합니다!
  2. PR하기까지 배움은 반드시 존재한다.
    • PR은 PR 내용에 확신이 있어야 가능하다고 생각합니다. 확신이 들기 까지 코드 및 로직은 무조건 이해를 해야만 합니다. 그 과정에서 개인적인 공부가 정말 많이 되었습니다. 다른 사람이 작성한 코드를 읽고 이해하는 과정에서 크고 작은 배움는 반드시 있다고 생각합니다.
  3. 오픈소스 생태계를 대하는 마음가짐이 달라졌다.
    • 오픈소스 생태계에 한 발짝 다가가게 된 것 같습니다. ‘누군가가 만든것을 사용’ 하는 것이 아닌 ‘우리 모두가 만들어갈 수 있는 것’ 이라는 생각이 생겼습니다.

마치며

누군가에겐 별 것 아닌 일일 수 있지만, 제겐 소중한 첫 기여 경험이었습니다! 오픈소스를 사용할 때는 더 많은 관심과 책임감을 가지고 개발할 수 있게 된 것 같습니다.

오픈소스에 기여해보고 싶은 개발자라면, 기여에 목적을 두기보단 사용하고 있는 오픈소스에 디테일하게 관심을 가지면 자연스럽게 기여하게 될 것이라고 말해주고 싶습니다.

긴 글 읽어주셔서 감사합니다!
